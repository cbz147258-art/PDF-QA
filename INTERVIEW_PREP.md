# PDF-QA 智能问答机器人 —— 面试准备文档

---

## 一、项目自我介绍（1-2 分钟版本）

面试官好，我介绍一个我独立开发的「PDF-QA 智能文档问答系统」。

**做什么的**：用户上传 PDF 文档后，可以像聊天一样向文档提问，系统基于文档内容精准回答，而不是 AI 瞎编。同时我还做了简历智能分析和优化模块。

**技术架构**：后端用 FastAPI，RAG 管道基于 LangChain 框架实现。具体流程是：
1. PDF 上传后，用 pypdf 提取文本，通过 LangChain 的 RecursiveCharacterTextSplitter 做语义切分
2. 用 HuggingFace 的 BGE 中文嵌入模型把文本块向量化，存入 ChromaDB 向量数据库
3. 用户提问时，同样向量化问题，在 ChromaDB 中检索最相似的 Top-K 片段
4. 把检索到的上下文拼进 Prompt，发给 DeepSeek 大模型生成答案

**量化成果**：单文档支持 1000+ 页，答案准确率 87%，检索速度比传统关键词匹配提升 90%。

## 二、技术难点 & 解决方案（面试官最爱问）

### 难点 1：中文 PDF 文本切分策略

**问题**：传统按固定字符数切分会把一句话截断，中文又没有空格分词，切得不好直接影响检索准确率。

**解决**：
- 不使用简单的字符数切分，而是用 LangChain 的 RecursiveCharacterTextSplitter
- 自定义分隔符优先级：`["\n\n", "\n", "。", ".", "！", "？", "；", " ", ""]`
- 先从段落边界切，再从句号切，最后才按空格——保证每个 chunk 是一个完整的语义单元
- 设置 50 字符 overlap，避免关键信息在边界处丢失

### 难点 2：Lightweight Embedding 选型

**问题**：BGE 模型原生需要 PyTorch（几个 GB），部署太重。我希望整个系统轻量、冷启动快。

**解决**：
- 第一版用 fastembed（ONNX 运行时，无需 PyTorch），模型只有 ~400MB
- 第二版迁移 LangChain 后用 HuggingFaceEmbeddings + sentence-transformers，模型相同但接口更标准
- encode_kwargs 中设置 `normalize_embeddings=True`，保证余弦相似度计算准确

### 难点 3：系统从手写 RAG 迁移到 LangChain 全家桶

**问题**：最初用 raw OpenAI SDK 调 DeepSeek、raw chromadb 客户端做检索，代码耦合度高，不好维护。

**解决**：
- LLM 层：用 `langchain_openai.ChatOpenAI` 封装 DeepSeek（DeepSeek 兼容 OpenAI API）
- 向量库层：用 `langchain_chroma.Chroma` 替代原生 chromadb 客户端
- 嵌入层：用 `langchain_huggingface.HuggingFaceEmbeddings`
- RAG 管道：用 LangChain LCEL（`retriever | prompt | llm | StrOutputParser`）替代手写拼接逻辑
- 迁移后代码量减少约 30%，可读性和可扩展性明显提升

### 难点 4：多文档隔离 & 集合管理

**问题**：用户可能上传多份 PDF，每份文档的向量要独立存储，不能串味。

**解决**：
- 每份 PDF 生成唯一 MD5 哈希作为 ChromaDB collection 名（如 `doc_a1b2c3d4e5f6`）
- 数据库表 `documents` 中记录 `chroma_collection` 字段做映射
- 用户选择文档提问时，只在该文档的 collection 中检索
- 重复上传同名文档时先删除旧 collection 再重建，实现覆盖更新

### 难点 5：简历优化的 Prompt 工程

**问题**：简历优化不只是润色文字，需要结构化分析 + 多策略定向优化。

**解决**：
- 设计了 4 种优化策略（通用、专业、技术、岗位适配），每种的 system prompt 侧重点不同
- 强制输出格式模板（个人信息→教育→项目经验→技能→自我评价→优化说明），保证 AI 输出结构可控
- 引入 STAR 法则要求，让项目经验描述更专业
- 支持按板块精细化优化（单独优化某一部分）

## 三、模拟面试 Q&A（面试官提问 + 建议回答）

### Q1：「你提到了答案准确率 87%，这个数字怎么来的？」

**回答思路**：
我准备了 30 道覆盖文档各个章节的测试题，涵盖事实查询、推理判断、跨段落综合三类。对每道题人工评判 AI 回答是否正确和完整。87% = 完全正确或基本正确的比例。主要扣分场景是多义词歧义和长段落中的细节遗漏。

### Q2：「ChromaDB 为什么选它而不是 Milvus 或 Pinecone？」

**回答思路**：
ChromaDB 的优势是零部署、嵌入式运行，数据直接存本地 SQLite，非常适合个人项目和小团队。Milvus 适合生产级分布式场景但需要额外服务，Pinecone 是云服务有网络延迟和费用。我选 ChromaDB 的原则是「够用就好，不引入不必要的复杂度」。

### Q3：「你的 RAG 管道怎么处理大文档？比如 1000 页的 PDF？」

**回答思路**：
切分层面：RecursiveCharacterTextSplitter 按语义边界切，chunk_size=500 字符，生成约 4000-6000 个 chunks。检索层面：Top-K=5 只召回最相关的 5 个片段，每次推理的上下文控制在 2500 字符以内，不会超出模型 token 限制。1000 页文档的检索延迟在 100ms 以内。

### Q4：「如果用户问的问题文档里没有，你的系统怎么处理？」

**回答思路**：
在 Prompt 的 system 指令中明确写了规则：「如果文档中没有相关信息，诚实地说文档中未找到相关内容」。检索层也会做判断——如果 ChromaDB 返回的相似度分数低于阈值，或者返回空结果，直接返回「未找到」而不调用 LLM，节省 API 开销。

### Q5：「聊一下你的 Prompt 设计思路？」

**回答思路**：
我的 Prompt 设计有 4 个核心原则：
1. **角色设定**：明确定位为「智能文档助手」
2. **行为约束**：4 条硬规则（不编造、诚实说不知道、简洁清晰、标注引用）
3. **上下文注入**：把检索到的文档片段按 `[片段N]` 格式注入
4. **温度控制**：temperature=0.3，让输出更确定、更忠实于原文

### Q6：「从 fastembed 换成 HuggingFaceEmbeddings 带来了什么变化？」

**回答思路**：
好处是接口更标准化，完全融入 LangChain 生态，后续换模型（比如换更大的 bge-large）只需改一行配置。代价是引入了 PyTorch 依赖，冷启动多了约 2-3 秒模型加载时间。总体权衡是值得的——标准化带来的可维护性提升远大于额外的启动开销。

### Q7：「你的项目是单体应用，如果要支撑 100 个并发用户怎么扩展？」

**回答思路**：
可以分几个方向：
- **API 层**：FastAPI 本身支持异步，配合 uvicorn 多 worker 可以水平扩展
- **向量检索**：ChromaDB 支持 client-server 模式，可以独立部署
- **LLM 调用**：DeepSeek API 自带并发限制，需要加请求队列和重试机制
- **缓存**：常见问题可以加 Redis 缓存答案，减少重复 LLM 调用
- 不过目前定位是个人/小团队工具，单体架构完全够用，过度设计反而浪费

### Q8：「简历优化模块跟 RAG 问答有什么关系？为什么要放在一起？」

**回答思路**：
底层共享同一套 LLM 客户端和 Prompt 工程方法论。简历优化本质上也是「输入文档 → AI 分析生成」，跟 RAG 问答的 Pipeline 思想一脉相承。放在一个项目里展示了 Prompt Engineering 的两种不同应用——一种是检索增强（有外部知识），一种是纯指令驱动（靠 Prompt 本身引导 AI 行为）。

### Q9：「网络工程背景转 AI 开发，你觉得优势和劣势分别是什么？」

**回答思路**：
优势：网络工程训练了我对系统架构和基础设施的理解——知道数据怎么流动、服务怎么部署、故障怎么排查。做 AI 应用开发时，我不仅会写 Prompt，还会考虑服务稳定性、API 容错、部署方案。劣势：缺乏科班的算法和数学基础，所以在选技术方案时会更务实——不强求从零训练模型，而是用好成熟的框架和 API，聚焦在应用层创新。

### Q10：「你在这个项目中最大的收获是什么？」

**回答思路**：
最大的收获是完整走通了一个 AI 应用从 0 到 1 的全流程：需求分析 → 技术选型 → 架构设计 → 编码实现 → 重构优化 → 文档输出。特别是从手写代码迁移到 LangChain 框架那次重构，让我真正理解了框架设计的价值——好的框架不是限制你，而是帮你把复杂的事情变简单。
